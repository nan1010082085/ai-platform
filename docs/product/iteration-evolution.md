## 一、现状诊断（产品视角）

### 1.1 产品定位

AI 项目是「可独立部署、可开源的 AI 应用能力小平台」，核心五大能力：

| 能力 | 成熟度 | 说明 |
|------|--------|------|
| AI 对话 | 高 | LangGraph 多专家 Agent，WebSocket 流式，多模态输入，@提及 + RAG 引用 |
| Agent 工作流 | 高 | n8n 风格 DAG 编排，16 个内置模板，发布后 REST/Webhook 调用 |
| RAG 知识库 | 中 | 文档向量化索引 + 检索增强，缺文档预处理深度 |
| 插件中心 | 高 | Expert/Skill/Tool/MCP 配置化扩展，热重载 |
| 对外集成 | 中 | API Key 调用已发布工作流，但集成密钥 UI 待建（platform.md 第七节承认） |

定位清晰，工程化程度高（strict TS、38 个单测、38 篇文档、Docker 一体化）。问题不在「能不能用」，而在「能不能开源、能不能增长、能不能持续吸引贡献」。

### 1.2 核心不足清单（按产品影响排序）

| 编号 | 不足 | 产品影响 | 根因 | 归属阶段 |
|------|------|----------|------|----------|
| A-01 | 无国际化（硬编码中文） | 开源后无法出海，海外用户无法使用 | 全部 UI 文案硬编码中文，仅 ElConfigProvider 设 zh-CN | A1 立即启动 |
| A-02 | 无前端埋点 | 开源后无法度量用户行为，增长两眼一抹黑 | 仅有后端 AI 调用监控（AiMonitorView），前端零埋点 | A2 |
| A-03 | 长对话无虚拟列表 | 核心对话场景超长会话卡顿 | 消息线性渲染，无虚拟滚动 | A1 |
| A-04 | 文件上传/下载逻辑重复 | 维护成本，新增文件操作易遗漏错误处理 | aiApi.ts 中 uploadFile/uploadRagDocument/downloadDocumentFile/downloadConversation 重复实现 fetch+buildHeaders+blob 模式 | A1 |
| A-05 | 测试 setup 残留 TDesign | 误导新人以为项目混用两套 UI 库 | 项目已迁 Element Plus，但 __tests__/setup.ts 仍 stub t-button/t-dialog | A1 立即修 |
| A-06 | 文档过时引用 | 新人按文档找不到目录，外部贡献者受阻 | README 提到 ai/sdk（实际不存在）、ai/shared（实际在 shared/platform-shared/ai/） | A1 立即修 |
| A-07 | 大文件待拆分 | 维护成本高，首屏偏重 | ai.ts 18KB、AiPreviewPanel.vue 28KB、AiChatPanel.vue 23KB、AiMonitorView.vue 24KB | A2 |
| A-08 | 前端体验无监控 | 用户遇到错误无上报，问题被动发现 | 全局错误仅 console，无 Sentry/自研上报 | A2 |
| A-09 | 工作流模板缺行业场景化 | 16 个模板偏通用，难以直接打动垂直行业客户 | 模板以文档摘要/合同提取/FAQ 等通用场景为主 | A3 |
| A-10 | 集成密钥 UI 待建 | 对外集成能力不闭环 | platform.md 承认「集成密钥 UI 待建」「后端按 createdBy 过滤待收紧」 | A2 |
| A-11 | 预留事件类型未实现 | 架构超前于实现，易让使用者困惑 | thinker_*、quality_check_* 事件已定义但图节点未实现 | A2 |
| A-12 | coverage 目录空 | 覆盖率未度量，测试投入无依据 | 覆盖率报告未生成或未配置 | A1 |

### 1.3 成熟度评估

| 维度 | 评分（5 分制） | 说明 |
|------|----------------|------|
| 架构设计 | 4.5 | 双引擎（Chat LangGraph + Workflow DAG）、Orchestrator Store 编排、注册表渲染 |
| 工程化 | 4.0 | strict TS（仅 11 处 any）、38 单测、38 篇文档、Docker 一体化 |
| 功能完整度 | 4.0 | 对话/工作流/插件成熟，RAG 与集成密钥待补 |
| 用户体验 | 3.0 | 长对话无虚拟化、无埋点优化依据 |
| 开源就绪度 | 2.5 | 无 i18n、文档过时、测试残留误导项 |
| 可观测性 | 2.5 | 后端 AI 调用监控完整，前端体验监控缺失 |

**总体判断**：架构与功能是强项，开源就绪度与前端可观测性是短板。当前是「能力领先但开源基础设施滞后」的状态——东西好用，但外人想用/贡献时摩擦大。

## 二、演进目标

### 2.1 北极星

做一个开源、可独立部署、能持续吸引外部贡献的 AI 应用平台，让团队能用对话或可视化编排搭建专属 AI 工作流，并通过 API 对接外部系统。

### 2.2 三阶段目标

| 阶段 | 目标 | 衡量标准（可验收） |
|------|------|---------------------|
| A1 开源就绪 | 让代码「可被外部使用」 | 文档零过时引用；中文文案全部可替换（i18n 框架落地）；测试 setup 无误导；长对话不卡 |
| A2 体验与增长 | 让开源后「能度量、能增长」 | 前端埋点上线；错误监控上报；集成密钥 UI 闭环；核心大文件拆分 |
| A3 开放平台 | 让外部能「贡献工作流与插件」 | 行业场景模板包；第三方插件接入文档；市场雏形 |

## 三、分阶段进化路径

### Phase A1：开源就绪（让代码可被外部使用）

**目标**：消除外部使用与贡献的第一道摩擦。让新人按文档能跑通、让海外用户能用、让核心对话场景不卡。

| 编号 | 关键工作 | 验收标准 | 依赖 |
|------|----------|----------|------|
| A1.1 | 文档过时引用修复 | README 修正 ai/sdk、ai/shared 等路径；与实际目录结构一致；CI 校验文档内路径存在 | 无 |
| A1.2 | i18n 框架落地 | 引入 vue-i18n（或复用平台 i18n 能力）；全部硬编码中文抽取为 key；提供 zh-CN 完整语言包 + en-US 骨架；语言可切换 | 文案抽取 |
| A1.3 | 长对话虚拟列表 | 对话消息列表虚拟滚动；1000 条消息滚动 FPS ≥ 30；保留流式输出与滚动定位 | 无 |
| A1.4 | 文件上传/下载逻辑抽取 | 抽取通用 `requestBlob`/`uploadBlob` 高阶函数；aiApi.ts 4 处重复替换为调用；错误处理统一 | 无 |
| A1.5 | 测试 setup 清理 TDesign | 移除 __tests__/setup.ts 中 t-button/t-dialog 等 stub；确认全量测试通过 | 全量回归 |
| A1.6 | coverage 接入 | 配置 vitest 覆盖率报告；CI 卡核心模块覆盖率门槛（stores/api ≥ 70%） | 无 |
| A1.7 | 集成密钥 UI 补建 | ApiKeyManagerView 完善；platform.md 第七节「待建」项清零 | 后端接口对齐 |

**关键决策**：A1.2 i18n 是开源就绪的最大工作量。需决策语言包组织方式（按模块拆分 vs 单文件）与是否复用 platform-shared 的 i18n 能力（避免重复造轮子）。

**预期产出**：一个外部开发者按 README 能在 30 分钟内本地跑通、切换英文界面的可用版本。

### Phase A2：体验与增长（让开源后能度量与增长）

**目标**：建立可观测性，让产品迭代有数据依据；补齐对外集成的最后一公里；治理核心大文件。

| 编号 | 关键工作 | 验收标准 | 依赖 |
|------|----------|----------|------|
| A2.1 | 前端埋点接入 | 关键路径埋点（发对话/选模板/发布工作流/执行失败/插件启用）；埋点 dashboard 可见用户行为漏斗 | 平台侧上报通道 |
| A2.2 | 前端错误监控上报 | 全局错误捕获上报（Sentry 或对接平台监控）；WebSocket 断连/重试失败上报；不再仅 console | A2.1 通道 |
| A2.3 | 核心大文件拆分 | ai.ts 拆分（编排逻辑分散到子模块）；AiPreviewPanel/AiChatPanel/AiMonitorView 拆为子组件；单文件 < 15KB | 无 |
| A2.4 | 预留事件类型清理 | thinker_*、quality_check_* 要么实现图节点，要么从事件协议移除并文档说明 | 架构决策 |
| A2.5 | 后端 createdBy 过滤收紧 | 工作流/API Key 按 createdBy 严格过滤；多租户隔离回归测试通过 | server 侧 |
| A2.6 | 对话历史管理增强 | 对话搜索、标签、批量导出；长会话归档 | 无 |

**关键决策**：A2.4 预留事件类型是「实现 vs 移除」的取舍。建议按产品价值评估——若 thinker/quality_check 在 1 个版本内无明确落地场景，则移除并文档说明「未来以 X 方式回归」，避免架构债误导使用者。

### Phase A3：开放平台（让外部贡献工作流与插件）

**目标**：从「自建插件」到「开放生态」。让外部能贡献行业工作流模板与插件，支撑平台的长期生命力。

| 编号 | 关键工作 | 验收标准 | 依赖 |
|------|----------|----------|------|
| A3.1 | 行业场景模板包 | 新增垂直行业模板包（如客服/法务/研报），每包 ≥ 3 个开箱即用工作流 | 模板设计 |
| A3.2 | 第三方插件接入文档 | 完整 Expert/Skill/Tool/MCP 开发指南，含脚手架、API 参考、示例插件 | 无 |
| A3.3 | 插件市场雏形 | PluginMarketView 升级为可安装入口；支持从外部源加载插件包 | 安全审计 |
| A3.4 | 工作流模板分享机制 | 用户可将自建工作流发布为社区模板；其他用户一键安装 | A3.1 |

**关键决策**：A3.3 插件市场涉及外部代码加载的安全风险。必须先做安全审计（沙箱隔离、权限声明、来源校验），参考 expert-manager 的安全检查流程。

## 四、关键决策点

| 决策 | 背景 | 选项 | 倾向 |
|------|------|------|------|
| D1 i18n 落地方式 | 开源就绪最大工作 | A 全量一次性 / B 按模块渐进 / C 仅搭框架延后 | B（核心交互优先，设置页延后） |
| D2 i18n 复用 | 避免重复造轮子 | A 复用 platform-shared i18n / B 自建 vue-i18n | A（与 editor/flow 统一） |
| D3 监控自研 vs 三方 | 上报通道 | A Sentry / B 对接平台已有监控 / C 先 console 增强 | B（与 editor 统一，避免多套） |
| D4 预留事件类型 | 架构债 | A 实现图节点 / B 移除并文档说明 | 评估 1 版本内有无场景，无则 B |
| D5 模板行业方向 | 资源有限 | 客服 / 法务 / 研报 / 通用增强 | 先做用户调研，选 1 个垂直深入 |
| D6 工作流分享安全 | 外部模板加载 | A 全开放 / B 平台审核白名单 | B（审核制，安全优先） |

## 五、成功度量

### 5.1 A1 完成标准（硬性）

- [ ] 文档内所有路径引用真实存在（CI 校验）
- [ ] i18n 框架落地，可切换英文界面（核心交互覆盖）
- [ ] 1000 条对话消息滚动 FPS ≥ 30
- [ ] 文件上传/下载逻辑无重复（通用函数抽取）
- [ ] 测试 setup 无 TDesign 残留，全量测试通过
- [ ] coverage 报告生成，核心模块门槛生效
- [ ] 集成密钥 UI 可用，platform.md「待建」项清零

### 5.2 A2 完成标准

- [ ] 前端埋点覆盖 5 个关键路径，dashboard 上线
- [ ] 错误监控上报生效（不再仅 console）
- [ ] 核心大文件单文件 < 15KB
- [ ] 预留事件类型决策落地（实现或移除）
- [ ] 后端 createdBy 过滤回归测试通过

### 5.3 A3 完成标准

- [ ] 至少 1 个行业模板包发布（≥ 3 个工作流）
- [ ] 第三方插件开发指南完成，含可运行脚手架
- [ ] 插件市场可安装外部插件（含安全审计）

### 5.4 长期产品指标（A3 之后持续跟踪）

| 指标 | 当前 | 目标 | 度量方式 |
|------|------|------|----------|
| 对话消息渲染上限 | 未知（卡顿） | ≥ 1000 | 性能测试 |
| 开源后首跑通时长 | 未知 | ≤ 30 分钟 | 用户调研 |
| i18n 语言数 | 1（中文） | ≥ 2（中/英） | 语言包数 |
| 前端关键路径埋点覆盖 | 0 | 5+ | 埋点 |
| 第三方贡献插件数 | 0 | ≥ 3 | 市场统计 |
| 工作流模板数 | 16（通用） | 16 + 行业包 | 模板库 |

## 六、风险与依赖

| 风险 | 影响 | 缓解 |
|------|------|------|
| A1.2 i18n 工作量大 | 阻塞开源就绪 | 渐进推进，核心交互优先，设置页延后；复用 platform-shared 能力 |
| A1.3 虚拟列表与流式输出冲突 | 滚动定位、自动滚底逻辑回归 | 专项测试流式追加 + 虚拟滚动组合场景 |
| A2.1 埋点需平台侧上报通道 | 跨项目依赖 | 早期与 server 侧对齐接口，先用 console 增强过渡 |
| A3.3 插件市场外部代码加载 | 安全风险（恶意插件） | 沙箱隔离 + 权限声明 + 来源校验 + 审核制 |
| A2.4 移除预留事件影响存量 | 协议破坏 | 评估存量引用，提供迁移说明 |
| 后端 createdBy 过滤 | 多租户隔离回归 | server 侧配合，前端测试覆盖 |
